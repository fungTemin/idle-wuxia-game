## 使用 AI Agent 构建全栈游戏项目的实践成果

我是一名后端开发工程师，平时主要写 Java，对前端 React 只有基础了解，从未独立完成过一个完整的游戏项目。这次我尝试用 AI 编程 Agent 从零搭建一个放置类游戏，整个过程让我对 AI 辅助开发有了全新的认知。

**项目起点只是一句话：「我想做一个放置游戏」。** 我没有写任何设计文档，也没有画架构图，而是通过和 Agent 对话来确定需求。它问我想要什么主题、什么技术栈、什么 UI 风格、第一版做到什么程度，几轮下来，一份完整的技术方案就出来了——数据库表结构、API 接口、前端页面划分全部列好。这比我平时自己做需求分析快了至少一个量级，而且它的提问很有引导性，帮我考虑到了很多我原本会遗漏的细节，比如离线收益的上限设置、宠物进化链的设计、成就系统的进度追踪逻辑。

**编码阶段的感受是「快，但需要懂行的人把关」。** Agent 一次性生成了约 9000 行代码，包含后端完整的分层架构（Controller-Service-Mapper-Entity）和前端 6 个页面。我原本以为 AI 生成的代码会是玩具级别的，但实际上代码结构清晰、类型定义完备、异常处理也到位。不过中间遇到了几个实际问题：MyBatis-Plus 版本升级后 PaginationInnerInterceptor 被移除了、JWT 过滤器没有正确设置 Spring Security 的认证上下文、TypeScript 严格模式下有重复属性报错。这些问题 Agent 都自己发现并修复了，但如果我对 Spring Security 完全不了解，可能都意识不到 401 报错是认证链路的问题。所以我的体会是：**AI 能写代码，但你需要有能力验证它写的东西对不对。**

**UI 迭代是最让我印象深刻的部分。** 我先后提出了三次风格调整：第一次说要古风水墨，第二次说要黑白配色，第三次给了一个很详细的设计规范（宣纸质感、飞白笔触、墨雾边缘、朱砂印章）。每次改动涉及 CSS 变量体系重写、SVG 资源创建、6 个页面组件同步更新。如果是人工来做，光是保证 6 个页面的设计语言一致就够头疼的了。但 Agent 在单次对话中能保持全局视角，改了配色方案后所有页面都会同步更新，不会出现这个页面用了新风格、那个页面还是旧风格的情况。这种跨文件的一致性维护能力，是我认为 AI 目前最超出人力的部分。

**环境搭建和调试环节，Agent 也帮了不少忙。** 我的机器上没装 PostgreSQL，它检测到后主动问我用 Docker 还是 H2 先顶着，我选了 Docker，它就帮我拉镜像、启容器、配置连接。后端启动后它还主动用 curl 测了一遍 API，发现 JWT 认证有问题后自己定位到 SecurityConfig 修复。这些琐碎但必要的工作，如果我自己来，至少要花半天查文档和调试。

**最终 4 小时内，我得到了一个可以玩的游戏。** 有注册登录、有放置收益、有战斗系统、有宠物收集、有商店和成就。代码已经推到 GitHub，README 文档齐全，任何人 clone 下来按步骤就能跑起来。如果让我估算同等质量的人工开发工作量，大概需要 3 到 5 个工作日——这还是在我熟悉后端的前提下，如果算上前端 UI 和水墨风格的 CSS 调试，可能要更久。

**总结下来，AI Agent 在这个项目中最大的价值是三件事：** 第一是需求到代码的转化效率，对话几轮就能生成完整方案；第二是跨技术栈的全链路能力，后端 Java 和前端 TypeScript 它都能写，不用我再找前端同事帮忙；第三是 UI 迭代时的全局一致性维护，改一个设计语言能同步更新所有页面。但前提是使用者需要有足够的技术判断力，能识别 AI 生成代码中的潜在问题，否则就会陷入「代码能跑但不知道为什么报错」的困境。
